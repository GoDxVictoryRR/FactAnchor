# ADR-001: Asynchronous Claim Verification Strategy

**Status:** Accepted

## Decision
Use Celery with RabbitMQ (not asyncio alone) for claim verification workers.

## Context
Verifying 100+ factual claims requires parallel execution. Python's asyncio handles concurrency on a single thread; it is unsuitable when workers need to be distributed across multiple machines in production.

## Consequences
- Workers are horizontally scalable independently of the API server.
- RabbitMQ provides durable task queues — if a worker crashes, the task is requeued.
- pgBouncer handles DB connection pooling so 50 concurrent workers don't exhaust PostgreSQL's connection limit.
- Trade-off: operational overhead of running RabbitMQ and monitoring worker health.

## Rejected Alternatives
- **FastAPI BackgroundTasks:** Not distributed.
- **asyncio gather:** Single-process, no retry.
- **AWS SQS:** Cloud lock-in for an on-prem enterprise target.
